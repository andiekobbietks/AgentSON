# Issue: AgentSON GDPR Compliance Model — Reconstruction Layer Positioning

> **RESOLVED** — 08 July 2026
>
> Implementation delivered in `cli/compliance.py` + `cli/reconstruct.py` + `spec/v1.2.json`.
> See decision log below for architectural choices.

## The core distinction

AgentSON is not a telemetry system. It does not sit inside the execution path of any tool. It is a **reconstruction layer** — it ingests partial, multi-system export artifacts and stitches them into a unified execution graph.

This distinction is critical for GDPR compliance claims:

| Claim type | Telemetry system | Reconstruction layer |
|---|---|---|
| Data origin | Direct instrumentation | Exported artifacts |
| Completeness | Full (by design) | Partial (by constraint) |
| Vendor cooperation | Required | Not required |
| Legal basis | Consent / legitimate interest | GDPR Art. 15/20 exercise |
| Guarantee | "Everything is captured" | "What is exportable is reconstructed" |

## The three-layer provenance model

Agent activity exists across three layers that no single vendor covers:

```
Layer 1: Communication (Slack, Teams, email)
  - Human-readable messages
  - File attachments
  - Edit history
  └── Exported via: Slack export, email export

Layer 2: Reasoning (Claude, ChatGPT, Copilot)
  - Assistant responses
  - Internal planning steps (where exposed)
  - Tool call invocations (where logged)
  └── Exported via: Chat history export, conversation JSON

Layer 3: Execution (CLI, IDE, CI/CD)
  - Commands run
  - Files changed
  - Errors and outputs
  └── Exported via: Shell history, IDE logs, git history
```

AgentSON's job is to **stitch these layers together** despite each being incomplete on its own.

## GDPR compliance boundaries

### What GDPR actually requires

Under GDPR Article 15 (right of access) and Article 20 (right to data portability):

| Data category | Exists in | GDPR obligation | Exportable via |
|---|---|---|---|
| Prompts you wrote | Layer 1/2 | Yes — data you provided | Slack export, chat history |
| Assistant responses you saw | Layer 2 | Yes — data generated by your interaction | Chat export |
| Tool outputs you observed | Layer 3 | Yes — data generated by your interaction | CLI logs, screenshots |
| Internal reasoning traces | Layer 2 | Usually no — proprietary | Not exported |
| System-level agent state | All | No — vendor's internal telemetry | Not exported |
| Other users' messages | Layer 1 | No — not your personal data | Excluded by law |

### What AgentSON can guarantee

AgentSON's compliance claims are bounded by what is **exportable at the source**:

```
✅ AgentSON CAN guarantee:
   • A standardized representation of all exported data
   • Temporal ordering of events within each source
   • Cross-source stitching where timestamps overlap
   • Transparent provenance tracking for each event
   • Machine-readable output in a portable format

❌ AgentSON CANNOT guarantee:
   • Complete coverage of all agent activity
   • Access to vendor-internal traces
   • Real-time capture (it's post-hoc)
   • Semantic understanding of reconstructed gaps
```

## Tractable vs. intractable reconstruction

Recent research establishes a formal boundary for what can be reconstructed post-hoc.

### Tractable: execution event sequences

**Execution events** (what commands ran, in what order, with what inputs/outputs, at what timestamps) are a **sequence reconstruction problem**. Multiple approaches from process mining and provenance analysis achieve strong results:

| Method | Domain | Result | Source |
|---|---|---|---|
| **SSSDS4 (diffusion)** | System call traces | Best accuracy across 9 datasets, various missing ratios | ICSE 2025 |
| **LSTM prefix+suffix** | Business process logs | Consistently outperforms baselines on missing activity labels | MDPI Information 2022 |
| **Heterogeneous GNN** | Event log attributes | Strong reconstruction of all event attributes | arXiv 2508.05145 |
| **LLM + trace similarity** | Process logs | Higher accuracy and robustness, with reflection to reduce hallucination | CIMS 2026 |
| **Particle smoothing (bi-LSTM)** | Continuous-time event streams | Effective at inferring unobserved events across synthetic and real domains | arXiv 1905.05570 |

These methods reconstruct **what happened** from partial event logs. They do not reconstruct **why**.

### Intractable: reasoning provenance

**Reasoning provenance** (why the agent chose each action, what it inferred from observations, which evidence shaped its strategy) **cannot** be faithfully reconstructed post-hoc in the general case.

The Agent Execution Record (AER) paper (arxiv 2603.21692) provides a formal non-identifiability argument:

> **Proposition.** Given only the persisted computational state $S_{1..K}$, reasoning provenance $R_{1..K}$ cannot in general be faithfully reconstructed as a normalized, schema-conforming, cross-run-comparable representation without contemporaneous capture at execution time.

Three sources of non-identifiability:
1. **Intent multiplicity** — different reasoning paths can produce identical state
2. **Observation compression** — what the agent observed is not fully recorded in state
3. **Epistemic asymmetry** — the agent's knowledge at decision time differs from the reconstructor's knowledge

### What this means for AgentSON

| Claim type | Reconstruction feasibility | AgentSON approach |
|---|---|---|
| "What events occurred" | Tractable | Pure stitching + optional SLM gap-filling |
| "In what order" | Tractable | Temporal + content-based stitching |
| "What was the input/output" | Tractable (if logged) | Source export + fingerprinting |
| "Why did the agent choose this" | **Intractable** | Not claimed — explicit gap marking |
| "What was the agent's internal state" | **Intractable** | Not claimed — vendor-internal |

AgentSON's gap-filling (when enabled) is bounded to execution event sequences — the tractable case. Reasoning reconstruction is explicitly out of scope per AER non-identifiability.

## The third path: live capture mode

The AER non-identifiability proof has a critical boundary condition: it applies to **post-hoc reconstruction from persisted state**. It does **not** apply to **contemporaneous capture at the communication layer**.

### How this works

Instead of reconstructing what happened after the fact, a **live listener** intercepts agent activity at the communication layer as it happens — the layer where the agent publishes its actions into a shared medium (Slack messages, IDE events, API calls).

For Claude Tag, this means a listener that:

1. Connects to Slack's Events API
2. Captures Claude Tag's messages (prompts, checklists, replies, files) as they arrive
3. Detects "Open session in Claude" links and scrapes the full tool-call trace from claude.ai
4. Converts everything to AgentSON events with `confirmed` provenance in real time

### Why this changes the compliance picture

| Aspect | Post-hoc reconstruction (forensic/narrative) | Live capture |
|---|---|---|
| When capture happens | After session ends | During session execution |
| Event provenance | `inferred` / `estimated` / `ml_generated` | `confirmed` |
| Reasoning visibility | Not reconstructible (AER proof) | Captured at communication layer |
| AER limitation | Applies in full | Sidestepped — contemporaneous |
| User prep needed | None (works on any historical data) | Must install listener before session |
| Dependency | Exported artifacts only | Slack API + extension + claude.ai session |

### The key insight

The AER non-identifiability proof proves you can't figure out *why* from *what was saved*. But with a listener, you're not reading what was saved — you're reading what **was published in real time** by the agent into a shared communication channel. If Claude Tag publishes its reasoning steps into Slack (checklists, thinking blocks, tool call results), those are captured as `confirmed` events, not reconstructed.

This is not telemetry (instrumenting the agent's internals). It's **communication surface intercept** — reading the same messages everyone in the channel sees, but capturing them in structured form.

### Relationship to the three modes

```
forensic     → post-hoc, gaps explicit (GDPR compliance default)
narrative    → post-hoc, gaps filled by SLM (opt-in, labeled)
live         → contemporaneous, no gaps (Claude Tag + extension)
```

Live capture is the strongest mode for trace fidelity, but requires infrastructure (listener + extension) that forensic/narrative do not. All three emit the same AgentSON format — only the provenance differs.

### Implemented in

The Claude Tag reader architecture is designed in `issues/issue-new-readers.md`, covering the browser extension, Slack API listener, and hybrid approaches.

## The stitching methodology

AgentSON uses three mechanisms to reconstruct execution from partial exports:

### 1. Temporal overlap

If Slack message at `T1` references a Claude response at `T1+2s`, and the Claude export contains a response at `T1+2s`, AgentSON links them.

### 2. Content fingerprinting

If a Slack attachment hash matches a file in the Claude export, AgentSON deduplicates and links.

### 3. Explicit instrumentation (opt-in)

If a user configures AgentSON CLI to capture shell history or IDE logs, those events fill gaps between export snapshots.

### 4. Generative gap-filling via neuro-symbolic SLM (opt-in, research-validated)

Recent research (HunterAgent, arXiv 2605.29269, May 2026) validates that an SLM can propose plausible intermediate events between confirmed anchors using a **Generator–Verifier architecture**:

```
Generator (SLM) → proposes candidate events for a gap
    ↓
Verifier (deterministic) → checks each against:
           • Temporal feasibility (does the timing work?)
           • Physical constraints (file paths, process lineages)
           • Available metadata (hashes, content fingerprints)
    ↓
Cost-bounded beam search → scores + ranks verified candidates
    ↓
Epistemic budget → halts gracefully if gap is too wide
```

Research results (HunterAgent):
- **86.1% F1** on trace reconstruction under partial observability
- **6.4% path-level hallucination** (vs 61.5% for unconstrained LLM)
- Hallucination cut achieved by **hard-pruning schema violations** and requiring **physical evidence grounding** — not by using a better language model
- Precision stays at **≥84% even under 70% log wiping**

Applied to AgentSON's domain: an SLM trained on AgentSON event sequences could generate plausible event candidates for gaps in the execution graph, with each candidate grounded against surviving metadata (timestamps, file hashes, process trees). The Generator-Verifier architecture is more important than model size — the verifier is the hallucination barrier.

### Confidence scoring

Each reconstructed edge carries a confidence level:

| Confidence | Meaning | Typical source |
|---|---|---|
| `confirmed` | Direct evidence from source | Slack message, Claude response |
| `inferred` | Stitched from overlapping sources | Temporal alignment |
| `estimated` | Reconstructed from heuristic | Gap-filling between known events |
| `ml_generated` | Generated by SLM, verified by deterministic checks | Neuro-symbolic gap-filling (opt-in) |
| `unknown` | No data available | Missing source |

## Formal compliance claims

AgentSON makes four bounded compliance claims:

### 1. Portability enablement (Art. 20)

> AgentSON provides a structured, commonly used, machine-readable format (JSON) for agent session data. It does not create the legal right — it makes the right exercisable.

### 2. Transparency support (EU AI Act Art. 52)

> AgentSON captures AI-generated content traces in a machine-readable format, enabling downstream transparency tooling. It is not itself a transparency compliance system.

### 3. Audit readiness (Art. 5(2))

> AgentSON preserves provenance metadata for each event, supporting accountability and audit workflows. It does not replace organizational record-keeping.

### 4. Reconstruction honesty (Art. 5(1)(d))

> AgentSON marks every event with its provenance confidence level. It does not fabricate data to fill gaps. Gaps are explicit in the output.

## Implications for CLI design

The `agentson reconstruct` command should surface this model:

```
USAGE
  agentson reconstruct <sources...> [--mode <mode>] [--output <file>]

MODES
  --mode forensic     Post-hoc stitching. Gaps explicit. Default.
  --mode narrative    +SLM gap-filling. Generated events labeled.
  --mode live         Contemporaneous capture via listener/extension.
  --mode both         Produce all available views.

SOURCES
  --slack-export <dir>       Slack workspace export
  --claude-export <dir>      Claude chat export
  --shell-history <file>     Shell history (bash, zsh, powershell)
  --git-log <dir>            Git repository
  --ason-file <file>         Existing AgentSON file (for merge)

LIVE MODE FLAGS
  --listen                  Start live listener (blocks)
  --slack-token <token>     Slack Events API bot token
  --session-url <url>       Scrape session page from claude.ai URL

NARRATIVE MODE FLAGS
  --model <name>             SLM to use for gap-filling
  --temperature <float>      Generation temperature (default: 0.3)
  --alternatives <int>       Number of alternative fill proposals (default: 3)
  --budget <float>           Epistemic budget — halt above this cost

OUTPUT BEHAVIOR
  • All events tagged with source provenance
  • Temporal stitching with confidence scoring
  • Explicit gap markers where sources don't overlap (forensic mode)
  • Generated gaps labeled with confidence="ml_generated" + model metadata (narrative mode)
  • Machine-readable JSON with human-readable summary
```

## Implementation Delivered

| Module | What It Does | Key Files |
|--------|-------------|-----------|
| **Compliance Engine** | Dual-track EU/UK compliance manifest generation, provenance confidence bounds, AI Act Art. 50(2) transparency marking, precedent case citations | `cli/compliance.py` |
| **Reconstructor** | Three-mode session reconstruction (forensic/narrative/live), deduplication, gap detection, multi-source stitching | `cli/reconstruct.py` |
| **Schema Update** | Added `_transparency`, `compliance`, `reconstruct_mode`, `digest`, `gap_ms` fields to v1.2 schema | `spec/v1.2.json` |
| **CLI Integration** | `agentson reconstruct` command with `--jurisdiction`, `--mode`, `--format`, `--no-ai-act-marks` flags | `cli/main.py` |

### Key Architectural Decisions

1. **Dual-track at type level, not runtime switch** — `Jurisdiction.EU` / `Jurisdiction.UK` / `Jurisdiction.DUAL` are enum values baked into the compliance manifest. The output format is identical; the legal basis strings differ. This avoids conditional code paths.

2. **Precedent cases hard-coded, not fetched live** — The four key cases (Meta/Hive, WhatsApp v EDPB, Brillen Rottler, UK Adequacy) are citation constants. These change slowly (years). Live fetching would add a network dependency and break offline-first.

3. **EU AI Act Art. 50(2) marking is `_transparency`, not `transparency`** — The underscore prefix signals this is a compliance annotation, not a user-facing field. It can be stripped for display or counted for audit.

4. **Provenance bounds are explicit, not buried** — The compliance manifest lists 5 bounded claims. Each has a `limitation` field. This is legally defensible: the user cannot claim AgentSON guarantees what it explicitly disclaims.

5. **`reconstruct_mode` at session level** — A session produced in 'forensic' mode cannot be re-sold as 'live'. The mode is locked into the stream-meta.

### Future-proofing for 10+ year horizon

| Horizon | Change Expected | How AgentSON Handles It |
|---------|----------------|-------------------------|
| 2026-2027 | EU AI Act Art. 50(2) enforcement begins | `_transparency` marking already compliant; Code of Practice refinement is additive |
